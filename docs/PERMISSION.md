## 3. B端可见C端用户字段边界

> 同一B端内,不同类别暴露字段不同,避免过度暴露。

| 类别 | 可见字段 | 不可见 |
|---|---|---|
| 浏览数据 | 昵称、浏览时间、时长、是否收藏、转发数 | 联系方式、身份证 |
| 加入用户 | 昵称、加入时间、已解锁的项目方联系方式(单向) | 身份证、其他B端数据 |
| 下级用户 | 昵称、推荐数、加入数 | 联系方式(除非互换)、身份证 |
| 资源广场 | 昵称、资源/需求描述、所属B端、收藏(公开) | 手机/微信/身份证 |

## 4. 资源广场隐私边界

### 4.1 公开字段白名单
- 昵称
- 资源/需求描述
- 所属B端
- 收藏项目(公开)

### 4.2 脱敏字段
- 手机号: 默认隐藏,互换授权后可见
- 微信: 默认隐藏,互换授权后可见
- 身份证: 仅 user_kyc 表,加密存储,仅风控/提现模块可访问

### 4.3 用户可见性设置
每条资源/资料可设:
- `public` 全平台可见
- `b_only` 仅指定 `visible_b_id` 对应 B 端内可见(不一定等于用户首次所属 B 端)
- `hidden` 隐藏

### 4.4 搜索范围
- 按地区/行业/资源关键词搜索
- 不暴露精确位置,仅地区级

## 5. 举报与封禁流程

### 5.1 举报
```
C端用户发现违规资料/资源
  → 提交举报(user_report: pending,附理由)
  → 平台审核
  → handled(处理:警告/封禁) / dismissed(驳回)
```

### 5.2 拉黑
```
用户A拉黑用户B
  → user_block 记录
  → B不可见A资料,B不可向A发起互换
  → 单向,不通知B
```

### 5.3 封禁
```
平台封禁违规用户
  → c_user.status = frozen
  → 资料下线,互换禁用,新加入禁用
  → 已结算佣金保留,未结算佣金冻结待处理
```

## 6. 敏感信息访问规则

### 6.1 user_kyc 访问
| 模块 | 权限 |
|---|---|
| 提现模块 | 读 real_name + 校验 verified=approved |
| 风控模块 | 全字段(含 id_card_enc 解密) |
| 普通业务查询 | **不可访问**,ORM层拦截 |
| 用户本人 | 仅可读自己,且不返回 id_card_enc |

### 6.2 联系方式访问
| 场景 | 规则 |
|---|---|
| C端查项目方联系方式 | 加入项目(付费)后解锁 |
| C端查其他C端微信 | 互换 approved 后互见 |
| B端查加入用户联系方式 | 加入用户单向解锁项目方联系方式,非互查 |

## 7. 多租户访问规则

### 7.1 租户表(带 b_id)
- 应用层 ORM 全局拦截 `WHERE b_id=?`
- DB层 PostgreSQL RLS 兜底
- B端仅能访问自己 `b_id` 数据

### 7.2 平台级表(不带 b_id)
- 不启用 b_id RLS
- 由权限矩阵 + 字段脱敏控制
- 资源广场全平台可见(白名单字段)

### 7.3 租户上下文信任
- 登录态: JWT claim 携带 `tenant_context.b_id`
- 服务端查 `user_tenant_relation` 解析当前用户与B端关系
- **不信任客户端 `X-B-ID` 头**(仅未登录公开页作入口线索)
